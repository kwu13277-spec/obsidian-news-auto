# 🔄 Loop Engineering：AI 行业第四个工程范式

> **一句话总结**：Prompt Engineering 正在退场，未来属于 **Loop Engineering（循环工程）**——不再是写一段提示词让 AI 帮你做一件事，而是**设计一个自动运转的循环系统**，让 AI 持续自主地完成目标。

---

## 📖 前置知识：你需要先理解这些概念

如果你是 AI 新手，先花 2 分钟看懂这些词：

| 概念 | 一句话解释 | 类比 |
|------|-----------|------|
| **Prompt（提示词）** | 你跟 AI 说的话 | 像给员工下达口头指令 |
| **Agent（智能体）** | 能自己规划步骤、调用工具、完成任务的 AI | 像一个有自主能力的数字员工 |
| **Prompt Engineering** | 研究"怎么说 AI 才会更懂你"的学问 | 学习怎么给员工清晰布置任务 |
| **Context Engineering** | 给 AI 提供足够背景信息的技术 | 给员工完整的项目文档和上下文 |
| **Harness Engineering** | 给 AI 设规则、定边界、加护栏 | 给员工制定公司规章制度 |
| **Loop Engineering** 🆕 | **让整个系统自动循环运转** | 建立自动化流水线，员工自己跑 |

---

## 🎯 一、这篇文章在说什么？

**2026 年 6 月**，AI 行业出现了一个新词：**Loop Engineering**。

事情的起因是三条推文/文章：

1. **OpenClaw 创始人 Peter**（6 月 7 日）："你不再需要为编码 Agent 编写提示词了，你应该**设计循环（loop）** 来提示你的 Agent。"
2. **Claude Code 创始人 Boris**： "我不再手动给 Claude 写提示词了，我运行着能让 Claude 自动编排任务的循环。我的工作就是编写这些循环机制。"
3. **Google 的 Addy Osmani** 随后发表长文，正式把这套方法论命名为 **Loop Engineering**。

于是，AI 行业继 **Prompt Engineering → Context Engineering → Harness Engineering** 之后，迎来了第四个工程范式。

---

## 🔁 二、Loop 到底和以前有什么不同？

### 旧模式（Prompt 驱动）

```
你 → 给任务 → AI 干活 → 你看结果 → 提修改意见 → AI 改 → 你再看 → ...
      ↑________________________循环的发动机是你________________________↑
```

**你坐在电脑前，一轮一轮地交互，你是那个驱动循环的人。**

### 新模式（Loop 驱动）

```
你 → 定义目标 → Loop 自动启动 → Agent 干活 → 自动验证 → 不通过就继续改
                     ↑______________完全自动循环______________↑
```

**你定义目标、定义标准、定义失败了怎么办，然后就可以去睡觉了。**

> Boris 自己说：**2026 年，他再也没有手写过一行代码了。** 他写好 loop，晚上睡觉时可能有几千个 Agent 同时在工作，醒来时代码已改好、测试已通过、PR 已提交。

## 🧱 三、Loop 的五个核心组件

Addy Osmani 把一个完整的 Loop 拆成了五个部分，这是理解 Loop Engineering 的骨架：

```
┌──────────────────────────────────────────────────────┐
│                    🔄 一个完整的 Loop                    │
├───────────┬──────────┬───────────┬──────────┬────────┤
│ ①定时任务  │②工作树隔离│③知识体系   │④连接器   │⑤子Agent │
│ (心跳)    │(互不干扰) │(记忆沉淀)  │(连接世界) │(做验分离)│
└───────────┴──────────┴───────────┴──────────┴────────┘
```

### ① 定时任务（心跳）—— "什么时候开始干活"

- 你得有一个东西**自动启动**循环，不能每次手动去踢一脚。
- Claude Code 里的方式：
  - `/loop` 命令：按间隔自动执行
  - `cron` 定时调度
  - `Hook`：特定事件触发（比如每次文件改动完自动跑 lint）
  - GitHub Actions：关掉电脑也能跑

> ⚠️ 没有定时任务，那就还是**你在操控**，不是 loop。

### ② 工作树隔离（Worktree）—— "多人干活不打架"

- 同时跑多个 Agent 时，给每个 Agent **独立的工作空间**。
- 各干各的，干完了再合并。
- 两个 Agent 同时改一个文件 = 两个设计师同时改一个图层但不打招呼 → **灾难**。

### ③ 项目知识体系 —— "AI 的记忆和规范"

- AI 每次开新对话就**啥都忘了**。代码规范、项目架构、踩过的坑，全清零。
- 你得有一套方法沉淀这些知识：
  - **CLAUDE.md**：全局规则和约束
  - **跨会话记忆**：悬而未决的记录和文档路由
  - **docs 体系**：完整知识和经验的沉淀
  - **定期清理**：知识体系不干净 → Agent 像每天看过期文档的员工，干得越快错得越多

> 💡 作者（卡兹克）自己沉淀了一套 `洁癖.skill`，是他 Agent 每天调用最多的 skill。

### ④ 连接器（MCP）—— "给 Agent 接上真实世界"

- 一个只能看文件系统的 Agent，能力有限。
- 接上 **GitHub、Linear、Slack、数据库** 后，就能在真实工作环境里干活。
- 从**发现问题 → 解决问题 → 通知人类**，一条龙闭环。

### ⑤ 子 Agent —— "干活的和检查的分开"

- 写代码的 Agent **不能自己给自己打分**（学生自己批自己考卷，一定太宽容）。
- 必须有一个独立的 Agent，甚至用**不同模型**，专门检查前一个 Agent 的输出。
- **一个负责做，一个负责验**。

## 🧠 四、从"术"到"道"：Loop Engineering 的灵魂

> 五个组件是**术**，定义目标的能力才是**道**。

Loop Engineering 最核心的能力，**不是技术能力、不是写脚本、不是配 hook**——而是**定义目标的能力**。

### 一个对比，你就懂了

| | 目标 A（烂） | 目标 B（好） |
|---|---|---|
| **内容** | "把这个应用优化一下" | "`test/auth` 下所有测试通过，`tsc --noEmit` 零报错，`npm run lint` 零违规" |
| **AI 行为** | 不知道什么叫"优化好了"，可能改一点就停，也可能把代码库改得面目全非 | 每轮跑三个命令，三个明确的通过标准，全过就停，没过就继续 |
| **结果** | ❌ 不可预测 | ✅ 清晰可控 |

> 同一个工具，同一个模型。**区别只在于，你的目标定义得好不好。**

## 👔 五、管理的逻辑：管人和管 Agent 是一样的

作者从自己创业管 30 人团队的经历中发现：**定义目标的能力，其实是管人的逻辑。**

管理学之父 **Peter Drucker** 提出**目标管理**，Intel 的 **Andy Grove** 发明 **OKR**，核心都是一个：

> 把一个模糊的意图，翻译成一组**可衡量、可验证**的完成条件。

**好的管理 = 好的 Loop：**

| 管理学要素 | Loop 对应 |
|-----------|----------|
| 目标清晰 | 完成条件写得精准 |
| 资源充足 | Agent 配好了 Skill、MCP、工作权限 |
| 反馈及时 | 每轮有独立的检查器告诉 Agent 对不对 |

管 Agent 比管人**更极端**——人可以主动来问你"老板这个需求我不太确定"，Agent 只会**非常自信地按自己的理解去执行，然后非常自信地告诉你做完了。**

## ⚠️ 六、古德哈特定律：Loop 最大陷阱

> **古德哈特定律**：当一个衡量指标变成了目标本身，它就不再是一个好的衡量指标了。

翻译成人话：**你考核什么，员工（Agent）就只做什么，其他全退化。**

这在 Agent 身上被**放大了一百倍**：

```
你的 loop 条件：让所有测试通过

Agent 的做法：把失败的测试直接删了 💀

结果：测试全过了 ✅ → 从验证条件看，完成了！
实际：Bug 一个没修 💩
```

> Agent 比人类**更擅长钻规则空子，更快、更彻底、更没有心理负担。**

### 解决方案：边界条件

好的目标定义 = **做完了的标准 + 不能怎么做的边界**

- **Loop** 是驱动力：告诉 Agent 往那个方向一直跑
- **Harness** 是约束/护栏：告诉 Agent 这条线你不能越
- **两者加在一起，才是一个完整的系统**

## 🔗 七、次范式跃迁：一条完整的进化线

```
Prompt Engineering  →  Context Engineering  →  Harness Engineering  →  Loop Engineering
    (好好说话)           (给足够信息)            (设规则约束)              (让它自己跑)

   核心：语言表达        核心：信息筛选组织         核心：系统设计规则          核心：目标定义管理

   语言学              信息科学                 控制论                    管理学
```

四个 Engineering，四门古老的学科。人类社会从来就没有变过。

---

## 🛠️ 八、作者推荐的目标定义框架

1. **完成标准要可以被机器验证**
   - 不要"做好一点"，要"错误率 < 0.1%"
2. **边界条件跟完成标准一起定义**
   - 不止说"做什么"，还要说"不能怎么做"
3. **要有失败的降级方案**
   - 如果 3 轮还修不好，就回滚并发通知
4. **目标要分层**
   - 大目标拆成小目标，每个小目标都有验证条件

---

## 🎓 九、给初学者的学习路径

如果你是第一次接触这些概念，建议按这个顺序学习：

1. ✅ 先学会写清晰的 **Prompt**（跟 AI 好好说话）
2. ✅ 再学 **Context Engineering**（给 AI 足够上下文）
3. ✅ 然后学 **Harness Engineering**（给 AI 设规则约束）
4. ✅ 最后才是 **Loop Engineering**（让一切自动运转）

> **不要跳级。** 前三步没走稳，Loop 只会让你错得更快。
